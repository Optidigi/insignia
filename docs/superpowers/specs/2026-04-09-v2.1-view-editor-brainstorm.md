## Competitor Dashboard Deep Dive

### InkBay (Inkybay) — Print Shop Powerhouse
**Admin nav**: 8 sections — Dashboard, Products, Designs/Templates, Cliparts, Fonts, Printing Types, Orders, Settings.  
**Setup**: 5-step wizard within a single page (create → categorize → configure attrs → print area → publish).  
**Pricing**: Global rules engine attached to **Printing Types** (decoration methods), NOT per-product. Options: per-color, per-sq-inch, size range, quantity discounts, setup fees. Products inherit pricing by selecting which printing types apply.  
**Zone editor**: Boundary-box drag with optional PNG mask upload. No Konva-style canvas.  
**Templates**: Pre-designed with granular restriction controls (lock dimensions, prevent removal, character limits).  
**Key insight**: Pricing as a global rules engine is their strongest architectural decision.

### Zakeke — Visual Commerce Platform
**Admin structure**: Product > Side(s) > Print Area(s) hierarchy.  
**Setup**: Select Shopify product → configure first side → Ruler Tool (calibrate real-world dimensions in cm/inches) → draw print area → repeat for sides → add variants.  
**Pricing**: Separate "Pricing Rules" system with condition-based logic (per color count, design size, character count, quantity breaks). Rules are global, not per-product.  
**Zone editor**: Ruler Tool + rectangle draw or PNG mask. Real-world dimension calibration is unique.  
**Key insight**: Ruler Tool enables accurate print-ready file generation. Side > Print Area hierarchy is intuitive.

### Customily — Template-First Architecture
**Runs as standalone web app** (not embedded in Shopify admin).  
**Admin nav** (left sidebar): Store, Templates, My Product Bases, Option Sets, Orders.  
**Core concept**: Templates ARE the product configuration. Three-entity model: Template + Option Set + Shopify Product.  
**Template wizard** (strict 3-step):  
1. **Print File Settings** — canvas size, format, color mode, DPI. Build design elements (text layers, upload zones, clipart). Production output only, no product images.  
2. **Preview Settings** — upload blank product photos, map design elements onto mockup images for customer-facing preview.  
3. **Review** — print file + preview side-by-side, interactive test.  
**Key architecture**: Separation of production output vs. customer preview as first-class concepts. "My Product Bases" = reusable library of product definitions with pre-defined print areas, decoupled from templates.  
**Pricing**: Lives outside Customily in Shopify's native variant system. Customily Option Sets handle personalization only (no price/SKU impact).  
**Key insight**: Template-first is fastest onboarding. "Product Bases" as reusable library is powerful. But 43% of negative reviews cite learning curve — three-entity model (Template + Option Set + Product) is too many concepts.

### Printful — Gold Standard for Simplicity
**Admin nav** (sidebar): Stores, Orders, Product Templates, File Library, Mockup Maker, Billing, Settings (7 items).  
**Setup**: 5-step wizard — Choose product → Design (in-dashboard editor with layers, 3000+ clipart, Getty integration) → Configure placement zones (tabs above mockup: front, back, sleeves, labels) → Select variants & preview (auto-generated mockups per color) → Set pricing & publish.  
**Pricing**: Inline with design. "Your cost: $12.50, Your price: $24.99, Your profit: $12.49."  
**Templates**: "Product Templates" is a first-class nav item — save design + placement + print settings as reusable preset, apply across products.  
**Per-variant artwork**: Different designs per color variant (black tee → white logo, white tee → black logo).  
**Key insight**: Linear wizard wins for first-time setup. "Product Templates" as first-class concept. Not applicable to custom decoration shops (Insignia's market).

### Printify — Single-Page Consolidated
**Setup**: 6-step flow but **all in a single view** with side navigation toolbar (no multi-page wizard). Browse catalog → Upload/create design (AI image generator, Shutterstock integration) → Adjust print areas with drag-and-drop + alignment grid → Select color variants (auto-mockups) → Set pricing inline → Publish.  
**Templates**: Save layout (design + placement), apply cross-product. Ready-made seasonal templates available.  
**Key difference**: Consolidates editing, pricing, and publishing into one view — side nav switches between sections without reloading.  
**Bulk**: Duplicate function in dashboard + public API for true scale.

### Cross-Competitor Pattern Table

| Aspect | InkBay | Zakeke | Customily | Printful/Printify | **Insignia** |
|--------|--------|--------|-----------|-------------------|-------------|
| Setup model | Multi-page CRUD | Stepped wizard | Template-first | Linear wizard | Multi-page CRUD |
| Pricing location | Global rules engine | Global rules engine | Per-template | Inline with design | Split (method + view editor) |
| Zone editor | Drag-dot handles | Ruler tool + draw | Auto from product | Predefined zones | Konva canvas (richest) |
| Template/clone | Pre-designed templates | Templates with rules | Core architecture | Template library | Duplicate config only |
| Variant images | Per-variant upload | Auto-pull from store | Per-template | Auto-generated mockups | Per-variant upload |
| Nav sections | 8+ | 4-5 | 3-4 | All-in-one | 6+ |

### Critical Findings

1. **Insignia has the richest zone editor** (Konva canvas). Genuine competitive advantage.
2. **Nobody puts pricing inside the zone editor.** Insignia is the only one. All competitors separate it.
3. **Pricing should be a global rules engine** (InkBay + Zakeke pattern), not per-product config.
4. **Template/clone is table stakes.** Every competitor has it.
5. **Linear wizard wins for onboarding** but custom decoration market needs good editing UX too.
6. **Zakeke's Ruler Tool** for real-world dimensions is a smart idea to consider.

## Research Summary

### Merchant Behavior (from competitor analysis, forum research, persona modeling)

- Typical small custom shop: 30-80 base products, 4-12 color variants each
- Most common placements: left chest + full back (2-4 zones covers 80%+ of orders)
- Configuration is mostly **set-and-forget** after initial setup
- **#1 return task is price changes** (seasonal, cost increases) — not repositioning zones
- Merchants think "left chest embroidery = $8" as **one concept** — placement + pricing are unified in their mental model
- **Positioning on an image** is a different cognitive task from **setting prices** — done at different times, different mental modes
- Every competitor that exposes internal data models as navigation (separate pages for methods, views, placements, configs) gets complaints about setup difficulty

### Competitor Insights

| Tool | Key Pattern | Best Idea | Worst Pattern |
|------|------------|-----------|---------------|
| Printful/Printify | Single guided flow: pick → design → price → publish | Linear wizard, no multi-page CRUD | Only works for their network |
| Zakeke | Product > Side > Print Area hierarchy | "Side > print area" is intuitive; pricing rules as separate engine | Creates extra Shopify products |
| Customily | Template-first architecture | Auto-mapping print areas to products | Steep learning curve for advanced features |
| InkBay | Multi-page CRUD (mirrors Insignia) | Live pricing that updates with every click | Kitchen-sink admin, 7+ top-level sections |
| CustomInk | Single-flow design experience | Zones emerge contextually from product type | Consumer-facing, not admin |

### Key Principle

> Navigation should follow **merchant tasks**, not database entities.
>
> Merchants think: "set up this product" → "make it look right" → "check my orders"  
> Not: "configure decoration methods" → "create views" → "define placements"

## What To Keep, Kill, and Add

### KILL: Quick Start Presets

Generic pixel-position guesses on product images we've never seen. A "left chest" preset assumes specific photo framing that's wrong for almost every merchant's actual photos. The merchant still has to drag the zone — the preset saves one drag at best. Replace with "Clone from product" (see below).

### KILL: Shared Zones Toggle

Unnecessary decision point. The physical placement zone is the same spot on the garment regardless of color. Edge case so rare it doesn't justify a toggle every merchant has to understand. Default to shared, remove the toggle.

### KEEP + IMPROVE: Tabbed Right Panel

Pricing and placement are one concept in merchants' minds. Two tabs keep visual work clean while putting business config one click away:

- **Position** tab — zone list, position readout, clone-from-product, manage images link
- **Pricing** tab — base price, size tiers, default size, hide-when-zero, pricing summary

### ADD: Config Cloning Across Products (highest leverage)

The #1 gap. Typical merchant configures one polo, needs the same setup on 14 more. Today "Duplicate existing setup" creates a whole new config. What's missing: **"Apply this config's zones and pricing to these other products"** as a bulk operation. Would cut setup time by 80% for typical merchants.

Infrastructure already exists: `product-configs.server.ts` → `duplicateProductConfig()`.

### ADD: Live Storefront Preview

Today "Preview on store" just links to the product page. Merchants never see the customer modal until they open the storefront. An inline preview panel/modal showing the actual customization experience with a sample logo would:
- Reduce setup anxiety
- Cut support tickets
- Give instant feedback during zone positioning

Best if it lives inside the View Editor as a third tab.

### ADD: Pricing Summary/Calculator

Merchants set method base prices + placement fees + tier adjustments but never see the total. A widget showing full breakdown (method base + placement base + tier adj = total) prevents pricing mistakes. Lives in the Pricing tab.

### REPLACE: Quick Start Presets → Clone From Product

Instead of 4 generic presets, a "Clone layout from..." dropdown of existing product configs. Real positions from real configurations, not guesses. Matches merchant mental model: "this polo is laid out the same as that polo."

More useful than user-created preset templates (which add naming/organizing/updating burden).

## Revised View Editor Layout

### Left side (canvas)
- View tabs (Front, Back, Sleeve) across top
- Konva canvas with product image and draggable zones
- Variant bar at bottom (color pills)
- No shared zones toggle (always shared)

### Right panel — Position tab
- Zone list (click to select on canvas, position readout)
- "Clone layout from..." dropdown (replaces presets)
- Manage product images link

### Right panel — Pricing tab
- Per-zone: base price, size tiers, default size, hide-when-zero
- Pricing summary widget ("Customer pays: €11.00 for Left Chest Embroidery at Medium")

### Right panel — Preview tab (future)
- Mini storefront preview of the customer modal
- Updates live as zones/pricing change
- "Open full preview" button

**Three tabs. Three tasks. Position it, price it, preview it.**

## Applicability Analysis — What From Competitors Applies to Insignia

### Context: What Makes Insignia Different

Insignia is **not** a print-on-demand fulfillment service (Printful/Printify), **not** a full design studio (InkBay), and **not** a multi-platform personalization engine (Customily). Insignia is a **logo placement configurator** — merchants define where logos can go and what they cost, customers upload logos and pick spots. This narrower scope means we shouldn't copy competitors wholesale — we should cherry-pick what maps to our actual use case.

Insignia is also **embedded in Shopify admin** (not standalone like Customily), which means we inherit Shopify's nav patterns and should follow Polaris conventions for consistency.

### Pattern-by-Pattern Assessment

#### 1. Global Pricing Rules Engine (InkBay + Zakeke)
**Pattern**: Pricing is configured once on decoration methods/printing types and applies globally. Products inherit pricing by selecting which methods apply.  
**Applicable?** Partially. Insignia already has `DecorationMethod.basePriceCents` (global) + per-placement `PlacementDefinition.basePriceAdjustmentCents` + per-step `PlacementStep.priceAdjustmentCents`. The architecture is actually **more granular** than InkBay/Zakeke — method base + placement adjustment + size tier. This is correct for Insignia's use case (pricing varies by zone AND size, not just by method).  
**Action**: Keep current pricing model. Don't flatten to method-only pricing. But **surface the full price calculation** to merchants (pricing summary widget).

#### 2. Separate Pricing From Zone Editor (All Competitors)
**Pattern**: No competitor puts pricing controls inside the visual zone editor.  
**Applicable?** Yes. This is the strongest signal from the research. Visual positioning and price configuration are different cognitive tasks.  
**Action**: Move pricing to a **Pricing tab** in the right panel (Option B). The Position tab stays purely visual.

#### 3. Linear Wizard for First Setup (Printful/Printify)
**Pattern**: Pick product → design → price → publish as one flow.  
**Applicable?** Partially. Insignia already has a "Create Setup" wizard modal with presets. The full Printful linear flow doesn't map because Insignia merchants bring their own products and production — there's no "pick from catalog" step.  
**Action**: Keep the Create Setup wizard. Enhance it by making the "Duplicate existing" option more prominent and powerful.

#### 4. Template/Clone as First-Class Feature (All Competitors)
**Pattern**: Every competitor makes templates or cloning a core feature. Printful has "Product Templates" in the main nav. Customily's entire architecture IS templates.  
**Applicable?** Yes, but fit to Insignia's model. Insignia's "ProductConfig" already IS a template — it bundles views, placements, pricing, and can be linked to multiple products. The missing piece is **applying an existing config's zone layout to a new config** (not just duplicating the whole config).  
**Action**: Add "Clone layout from..." in the View Editor Position tab. Enhance "Duplicate existing setup" in the Create wizard. Consider a "Manage templates" secondary view on the Product List page.

#### 5. Product Base Library (Customily)
**Pattern**: Reusable product definitions with pre-defined print areas, decoupled from individual product configs.  
**Applicable?** Not directly. Customily needs this because they separate print output from preview — two independent config layers that need a shared base. Insignia's ProductConfig already bundles everything. Adding a separate "product base" abstraction would add complexity for minimal gain.  
**Action**: Skip. Config cloning achieves the same outcome without adding concepts.

#### 6. Ruler Tool / Real-World Dimensions (Zakeke)
**Pattern**: Calibrate product image to real-world measurements (cm/inches) for accurate print-ready files.  
**Applicable?** Not yet. Insignia stores percentage-based geometry (centerXPercent, centerYPercent, maxWidthPercent) which is display-agnostic and works well for storefront rendering. Real-world dimensions matter for production file output — but Insignia doesn't currently generate print-ready files.  
**Action**: Skip for now. Revisit when/if production file export becomes a feature. Store real-world dimensions as optional metadata on PlacementDefinition if future-proofing is desired.

#### 7. Per-Variant Artwork Assignment (Printful/Printify)  
**Pattern**: Different designs per color variant within the same product.  
**Applicable?** Insignia already handles this. Per-variant images via VariantViewConfiguration, and customers upload their own logos per placement.  
**Action**: No change needed. Already handled.

#### 8. Auto-Pull Product Data From Shopify (Zakeke)
**Pattern**: Zakeke automatically pulls product images, variants, and attributes from the Shopify store.  
**Applicable?** Insignia already does this for variants (via GraphQL). Product images can be imported in the Image Manager.  
**Action**: Could improve by auto-suggesting image import when a product is first linked. Low priority.

#### 9. Reduce Nav Sections / Task-Based Navigation (All)
**Pattern**: Fewer top-level nav items, organized by tasks not entities. Printful: 7 items. Customily: 5. InkBay: 8 (criticized for it).  
**Applicable?** Yes. Insignia currently has: Home (dashboard), Products, Orders, Methods, Settings — that's actually only 5 top-level items, which is reasonable. The problem is the depth within Products: Product Detail → View Editor → Placement Editor → Image Manager = 4 levels deep.  
**Action**: The View Editor absorbing pricing (via tabs) reduces the need for a separate Placement Editor page. The Product Detail → View Editor becomes the primary drill-down, and within the View Editor merchants can do both position and pricing. This flattens the navigation depth from 4 levels to 2.

#### 10. Pricing Summary / Calculator (InkBay-inspired)
**Pattern**: InkBay shows live pricing that updates with every customer click. Printful shows cost vs. price vs. profit inline.  
**Applicable?** Yes. Merchants set prices across three levels (method + placement + tier) but never see the total. A simple widget: "Embroidery + Left Chest + Medium = €11.00" would prevent errors and build confidence.  
**Action**: Add to the Pricing tab of the View Editor right panel.

### Summary: What To Do

| # | Action | Source Pattern | Priority |
|---|--------|---------------|----------|
| 1 | Add Position/Pricing tabs to View Editor right panel | All competitors separate pricing from zones | **High** |
| 2 | Remove Quick Start Presets | Generic presets are noise (no competitor matches Insignia's use case) | **High** |
| 3 | Remove Shared Zones toggle | Edge case, adds unnecessary concept | **High** |
| 4 | Add "Clone layout from..." dropdown | Template/clone is table stakes (all competitors) | **High** |
| 5 | Add Pricing Summary widget | InkBay live pricing / Printful cost calculator | **Medium** |
| 6 | Enhance "Duplicate existing" in Create wizard | Customily template-first / Printful Product Templates | **Medium** |
| 7 | Add Live Storefront Preview tab | No competitor does this well — differentiator | **Low** (future) |
| 8 | Ruler Tool for real-world dimensions | Zakeke | **Low** (future, needs production file export) |
| 9 | Auto-suggest image import on product link | Zakeke auto-pull | **Low** |

### What NOT To Do

- **Don't flatten pricing to method-only.** Insignia's per-placement + per-tier pricing is more granular than InkBay/Zakeke and is correct for the use case.
- **Don't add a "Product Bases" abstraction.** Config cloning achieves the same thing without adding concepts. Customily's three-entity model is their biggest complaint.
- **Don't build a standalone web app.** Embedded Shopify admin is the right call — merchants live in Shopify admin.
- **Don't build a full design studio.** Insignia isn't InkBay. Customers upload logos, they don't design them. Keep the scope narrow.
- **Don't add per-printing-type pricing tables.** InkBay's per-color × per-quantity matrix is overkill for logo placement. Insignia's base + tier model is simpler and sufficient.

## Beyond the View Editor — Other UX Gaps Worth Considering

### Found via codebase audit + SaaS UX research

#### A. Dashboard Analytics Are Built But Never Rendered (quick win)
The loader in `app._index.tsx` already queries `analytics.methodBreakdown` (orders per decoration method), `needsAttention` (pending artwork), `recentConfigs`, and `activityEvents`. **The data is fetched but never displayed.** A method breakdown chart, pending-artwork count, and an activity feed are free wins — the backend work is done.

#### B. Production Status Workflow Has No UI
The schema defines `ProductionStatus`: ARTWORK_PENDING → ARTWORK_PROVIDED → IN_PRODUCTION → QUALITY_CHECK → SHIPPED. But the order detail page has no dropdown or buttons to advance orders through this pipeline. Merchants track production in their heads or in spreadsheets. A simple status stepper on the order detail page would fill this gap.

#### C. Bulk Operations Are Almost Nonexistent
Only batch image uploads exist. No bulk select for orders (multi-export, batch status update), no bulk delete/duplicate for product configs, no multi-method assignment. This forces one-at-a-time workflows for merchants managing dozens of products.

#### D. "Welcome Back" Experience
Merchants who return after weeks don't know what happened. A contextual dashboard banner — "3 new orders since your last visit, 1 pending artwork" — with deep links would re-orient returning merchants instantly. This addresses the #1 cause of churn in SaaS admin tools: "I forgot how this works."

#### E. Pricing Summary Across the App
Not just in the View Editor — the Product Detail page should show a pricing summary card: "Customer pays €X–€Y depending on placement and size." Currently merchants have to mentally calculate across method base + placement base + tier adjustment. This is the most common source of pricing mistakes.

#### F. Cmd+K Command Palette (power user feature)
The pattern that Linear, Notion, Vercel, and every loved SaaS tool uses. Search across products, orders, methods, settings by name. Especially valuable for returning merchants who forgot where things live. Low priority but high delight.

#### G. Toast + Undo for Destructive Actions
Currently deleting a placement, view, or method is permanent. No undo. Adding a 5-second toast with "Undo" for destructive actions is the safest pattern — merchants take bold actions because they know they can reverse them.

#### H. Inline Storefront Preview
Already captured in the View Editor recommendation (Preview tab). But worth emphasizing: **no competitor does this.** A live preview of the customer modal, updating as you move zones and change prices, would be a genuine differentiator. Merchants currently configure blind — they never see what customers see until they visit the storefront themselves.

### Priority Tiers

| Tier | Items | Rationale |
|------|-------|-----------|
| **Quick wins** (do alongside V2.1) | A (dashboard analytics), B (production status UI), E (pricing summary) | Backend already exists, UI is thin |
| **Medium priority** (next cycle) | C (bulk operations), D (welcome back), G (toast+undo) | Quality-of-life improvements that reduce support tickets |
| **Future polish** | F (Cmd+K), H (inline preview) | Delightful but not blocking |
